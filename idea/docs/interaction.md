# idea 系列交互约定

idea 系列的多个 skill 都会在运行过程中向用户提问——脑暴时抛反问、clarify 时给选项、conclusion / plan 在覆盖既有文件前征求确认、resume 在恢复后问"下一步走哪个 skill"。本文是这些**人机问答行为**的统一约定，与 [tag-system.md](tag-system.md) / [aliases.md](aliases.md) / [links.md](links.md) 平行——前三者管 frontmatter / wikilink，本文管"怎么向用户提问"。

idea-* skill 的 SKILL.md **不应**在本地重新定义编号格式或交互硬约束；它们只引用本文档，并在自身 `## 交互行为` 段补充本 skill 特有的差异化要求。

## 提问的两种形态

idea 系列里向用户提问归为两种，约束略有差别：

| 形态 | 典型 skill | 是否需要选项 | 是否需要推荐 |
| --- | --- | --- | --- |
| **决策提问**（让用户拍板） | `idea-clarify`、`idea-create`（冲突处理）、`idea-conclusion` / `idea-plan`（覆盖/追加/取消）、`idea-resume`（下一步走哪个 skill） | 必须 | 必须 |
| **挖掘反问**（不要求拍板，只挖用户的判断） | `idea-brainstorm` 各角度末尾的 `?` 行 | 可不带（开放式问题） | 不需要 |

两种形态的共同硬约束：**问任何问题之前先解释问题里的关键术语**（见下面"硬约束 1"）。

## 硬约束

下列五条对所有 idea-* skill 的提问环节都强制成立。

### 1. 提问前必须先解释关键术语

无论决策提问还是挖掘反问，问题里凡是出现可能被歧义理解的概念 / 名词 / 缩写 / 比喻，都要在抛出问题之前用 **1–2 句**说清楚"在本次提问里它指的是什么"。

目的：把用户和 AI 锚定在同一基准，避免用户基于不同理解给出答非所问的回答。

形式：

- 决策提问里：在「问题」之后、「选项」之前，单起一行写「术语：<X 在此指 ... ; Y 在此指 ...>」，多个术语用分号分隔
- 挖掘反问里：在 `? <反问>` 行之前，单起一行 blockquote 写 `> 关键词：<X 在此指 ... ; Y 在此指 ...>`

什么算"需要解释"：

- 跨领域借来的词（"涌现"、"耦合"、"摩擦力"等比喻）
- 本 idea 内部约定的命题特定词（idea.md 命题里出现过、但用户日常未必这么用）
- 容易被两种以上理解的常见词（"用户"指开发者还是终端读者；"成本"指时间、金钱还是注意力）
- 缩写（除非用户已经在 idea.md 里展开过）

什么算"不需要解释"：

- 字面就是字面、没有歧义可能的普通名词（"今天"、"列表"等）
- 本 skill 自身术语（如"反问"、"轮"），用户已通过本 skill 的 SKILL.md 知晓

不空喊"更通用"、"更灵活"——解释要落到本 idea 的具体语境。

### 2. 一次只问一个问题

决策提问永远**一次只抛一个**，等用户回答完再问下一个。禁止"问题 1 / 问题 2 / 问题 3 …"罗列；如果待问项有多条，先把"待确认清单"亮给用户审阅（让他可以删减 / 调整顺序），然后逐项进行。

挖掘反问允许在同一段（比如 brainstorm 一个角度内）放 1 条；不要在一个角度里堆叠多条 `?`。

### 3. 选项编号统一为 ABCD

决策提问的选项编号固定使用 **A / B / C / D**，最多 4 个。

- 一条决策最多 4 个选项；超过 4 个意味着维度还没拆清，先拆维度
- 如果某选项确有需要展开的子选项（典型如"用 `<name>-2` 命名"下还要决定具体后缀），用衍生格式 **A.1 / A.2**，仍不超过 A–D 四个一级
- 多选场景默认不开放；如确需多选，明确写「多选请回 `A,B`」并在 SKILL.md 自身的 `## 交互行为` 段说明
- "跳过" / "取消" 是元行为，**不进** ABCD 编号——用户回字面量「跳过」「取消」即可；任何决策提问都默认隐含支持这两个回应

示例（决策提问）：

```text
术语：「合并」在此指：把新设想追加到原 idea.md 末尾，而不是新建文件。

A. 用 `<name>-2` 等带后缀的新目录 — 与 B 的差异：保留两份独立工作区，互不污染；与 C 的差异：保留本次输入 — 后果：之后两个 idea 各自演化。
B. 合并到原目录的 `idea.md`（追加，不覆盖） — 差异：把新输入塞进既有命题；后果：单一工作区会变厚，可能混入两条本应独立的命题。
C. 取消本次创建 — 差异：什么都不写。

推荐：A
理由：根据 idea.md 命名约定，原目录已经有 ≥ 1 轮 brainstorm，新输入跟原命题角度不同，混在一起会污染原 brainstorm 的"下一轮焦点"。
```

### 4. 选项写法

每个选项必须同时写：

- **描述**：一行说清这条是什么
- **关键差异**：写「与 B 的差异：…」这种**对比关系**，不是把每个选项各自的优点列一遍
- **后果 / 取舍**：选这条会带来什么、放弃什么

差异要"说人话"——用一句具体场景说明它会怎样落到本 idea 上，不要只用抽象形容词（"更通用"、"更灵活"、"更轻量"等不算差异）。

### 5. 推荐 + 理由

每次决策提问必须**明确推荐一个选项**，且：

- 必须有偏好（不允许「都可以，看你」）
- 理由必须**带具体出处**——援引本 idea 内现有文件的某一条（例如「与 [[ideas/<idea-name>/brainstorm#第 2 轮]] · 反例与反驳里你提到的 X 一致」），禁止只说「更合适」
- 真的难分轩轾时，仍然选一个，并写「在 X 与 Y 之间偏向 X 的原因是 …」

## 用户回答的形式

允许的回答形式（决策提问通用）：

- 接受推荐：可回字面量「接受」、「同意」、「就推荐」，或直接回推荐项的编号
- 选别的：回编号 `A` / `B` / `C` / `D`，或描述性回答（例如「选第二个」）
- 自定义：用户给一个不在选项里的第 5 种答案——照实记录，不要硬塞回某个原选项
- 跳过：用户回「跳过」——按本 skill 的语义处理（例如 idea-clarify 把它放进"本轮未拍板"）
- 取消：用户回「取消」——本次决策不落，停在当前位置不写任何文件

skill 在用户**没有明确回答**前不允许替用户拍板，也不允许写入任何"假设用户选了 X" 的内容。

## 与 SKILL.md 的分工

- 本文档定义**所有 idea-* skill 共享的提问规则**（编号、术语解释、一次一题、选项写法、推荐要求、回答形式）
- 各 SKILL.md 在 `## 交互行为` 段：
  - 第一行写 `按 [docs/interaction.md](../../docs/interaction.md)：`，不重复抄本文规则
  - 之后只列**本 skill 特有**的交互差异，例如：
    - idea-clarify：跳过项进"本轮未拍板"列表
    - idea-brainstorm：每个角度恰挂 1 条挖掘反问，且反问要前置 `> 关键词：` 行
    - idea-create：冲突处理的 ABCD 选项是 `<name>-2 / 合并 / 取消`（不存在 D）
    - idea-resume：续接选项的 ABCD 由当前 workspace 的可用下一步动态生成
- SKILL.md **绝不**在自身写 "选项用 A/B/C 编号"、"一次只问一个" 这类已经被本文覆盖的规则——重复定义会让规则在两处漂移

## 兼容性

- 老版本 SKILL.md 没有 `## 交互行为` 段：在下次修改该 SKILL.md 时补；不强制一次性扫所有
- 用户回答里夹带额外信息（解释 / 反问 / 题外话）：skill 把决策落到「最终选择 + 理由」字段，用户额外信息可以放到模板的「备注」字段（如该 skill 的模板提供）
- 如本文规则与 SKILL.md 自身的"边界（强制）"段冲突：以"边界（强制）"为准（本文管交互行为，不能突破写文件边界）

## 与其他 docs 的关系

- 本文与 [tag-system.md](tag-system.md) / [aliases.md](aliases.md) / [links.md](links.md) 平行：那三份管 frontmatter 与 wikilink 写入；本文管向用户的提问方式。互不重叠
- skill 的"边界（强制）"段、"frontmatter / tag 行为"、"frontmatter / aliases 行为"、"链接行为" 仍由各 SKILL.md 自身保留，本文不接管它们
